# 已知局限性

GhostProver 作为本地合规 Agent 已具备实用价值，但以下限制对评审者和未来贡献者非常重要。

- **WASM 上的证明生成速度较慢。** 守护进程将证明任务放入后台队列，因此演示时不应等待整套预设证明全部完成。
- **模式长度固定。** 当前电路支持固定的 1–32 字节字符类型描述符，可选分隔符和可变长度格式是未来的工作方向。
- **尚不支持语义 PII 检测。** GhostProver 检查的是结构化模式，而非自然语言含义。
- **误报需要策略调优。** 长数字序列等宽泛模式在企业预设中应谨慎使用。
- **MCP 需要主动调用。** MCP 工具不会自动拦截所有 Prompt；Agent 工作流必须显式调用 GhostProver。
- **实时 0G 锚定为可选功能。** 守护进程的默认模式会写入草稿缓存记录以方便开发。主网提交需要将 `onChainSubmit` 设为 `true`，并准备好已部署的注册表合约和有余额的 `Compute/.env` 钱包。

## 推荐的后续技术改进

1. 为信用卡证明添加 Luhn 算法验证。
2. 实现可变长度模式编译器。
3. 为企业自定义策略提供注册表 UI。
4. 在 0G 适配器中复用守护进程已生成的证明，避免通过 Compute Orchestrator 重复证明。
5. 为守护进程访问添加生产级认证。
